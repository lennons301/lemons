name: feedback_next_steps
description: When summarizing next development phases, prioritize what the design doc actually specifies over assumptions
type: feedback
---

When asked about next steps, read the design doc thoroughly and present deferred/planned items that are explicitly called out — don't lead with assumptions about what sounds logical. The user caught that calendar + meal plan integration was redundant with the dashboard layout, while the actually-planned calendar + todo integration (event-linked lists) was buried lower in the list.

**Why:** The design doc is the source of truth for planned features. Guessing at priorities wastes the user's time.

**How to apply:** Always cross-reference deferred items in design docs against what's already built. Lead with what the design doc explicitly planned and deferred, not what seems like a natural next step.
